## 1. 规划闸门全局状态耦合

**位置**: [loop.ts:230-263](packages/core/src/agent/loop.ts), [todo.ts:50-91](packages/core/src/tools/builtin/todo.ts)

**原问题**: `AgentLoop.enforcePlanningGate()` 通过 `TodoWriteTool` 的静态方法（`hasActivePlan()`, `trackModifyCallWithoutPlan()` 等）访问规划状态，两者形成隐式强耦合。规划状态本应属于 Agent 的会话级状态，却耦合在工具类中。

**修复方案**:
- 创建 [plan-state.ts](packages/core/src/agent/plan-state.ts)，定义独立的 `PlanState` 类，持有 `todos`、`modifyCallsWithoutPlan`、`driftWarnings` 三个状态字段
- `AgentLoop` 在 `run()` 初始化时创建 `PlanState` 实例，通过 `TodoWriteTool.setPlanState()` 注入
- `TodoWriteTool` 的静态方法委托到注入的 `PlanState` 实例（未注入时使用内部默认实例，保持向后兼容）

**修复后架构**:
```
AgentLoop.run()
  ├── this.planState.reset()
  ├── TodoWriteTool.setPlanState(this.planState)  ← 注入
  └── enforcePlanningGate() → TodoWriteTool.hasActivePlan()
                                   → planState.hasActivePlan()  ← 委托
```

## 2. Token 估算截断优化

**位置**: [loop.ts:44-48](packages/core/src/agent/loop.ts), [loop.ts:792-818](packages/core/src/agent/loop.ts)

**原问题**: 
- 所有字符统一按 `length / 4` 估算 token 数，中文（实际 1-1.5 字符/token）误差可达 2-4 倍
- `slice(0, maxChars)` 盲目截断，可能在 JSON 中间或多字节 UTF-8 序列中间切断

**修复方案**:
- **区分字符类型估算**: ASCII（~4 字符/token）、CJK 统一汉字/平假名/片假名/韩文（~1.5 字符/token）
- **安全截断点**: 优先在换行符处截断，回退时跳过 UTF-16 低代理项（避免截断 emoji 等多字节字符）

**关键代码**: `estimateTokens()` 和 `findSafeTruncationPoint()` 方法

## 3. Bash 命令权限检查增强

**位置**: [manager.ts:392-401](packages/core/src/permission/manager.ts)

**原问题**: 
- `command.split(" ")[0]` 只取第一个空格前的词
- `npm ci && npm test` → `Bash:npm`（无法区分子命令）
- `sudo rm -rf /` → `Bash:sudo`（提取到提权工具而非实际命令）

**修复方案**: 新增 `extractBashBaseCommand()` 方法，按以下顺序解析：
1. 按 `;` / `&&` / `||` 拆分命令链，取第一段
2. 按 `|` 拆分管道，取第一段
3. 按 `<` / `>` 拆分重定向，取第一段
4. 按空白拆分为词，跳过提权前缀（`sudo`、`pkexec`、`doas`、`run0`）
5. 跳过 `env` 前缀及环境变量赋值（`KEY=value`）
6. 取第一个非空白词作为基准命令

## 4. 配置深度合并

**位置**: [loader.ts:251-268](packages/core/src/config/loader.ts)

**原问题**: 配置合并使用 `{ ...tgtVal, ...srcVal }` 浅合并。`providers.anthropic` 对象在 settings.json 和 settings.local.json 中分别定义时，后者会整体替换前者而非合并字段。

**修复方案**:
- 移除浅合并 `merge()` 方法
- 新增递归 `deepMerge()` 方法，对嵌套对象逐层合并，基础类型和数组直接覆盖
- 新增 `isPlainObject()` 辅助判断（排除数组和 null）

**影响**: `settings.local.json` 中可以只写需要覆盖的字段（如 `providers.anthropic.apiKey`），而不需要完整复制整个 `providers` 结构。

## 5. LLM 恢复计数修复

**位置**: [loop.ts:726](packages/core/src/agent/loop.ts)

**原问题**: `llmRecoveryCount` 在每次进入恢复流程时递增，但 `COMPACT_CONTEXT`（上下文压缩）成功后不重置计数器。压缩解决了根本问题（上下文溢出），但计数器未清零，导致后续如果 LLM 因其他原因失败时更快达到上限。

**修复方案**: 在 `handleLLMRecovery()` 中，`COMPACT_CONTEXT` 策略成功后立即重置 `this.llmRecoveryCount = 0`。

## 6. 权限 Cache Key 路径规范化

**位置**: [manager.ts:398-399](packages/core/src/permission/manager.ts)

**原问题**: Write/Edit 工具的缓存 key 使用原始 `file_path` 参数值，不做路径规范化。`./src/app.ts` 和 `src/app.ts` 被视为不同的 key，导致重复弹窗确认。

**修复方案**:
- 新增 `normalizeFilePath()` 方法
- Node 环境使用 `path.resolve()` 转为绝对路径
- 非 Node 环境（浏览器）回退：去除前导 `./` 和合并多余斜杠

## 7. 工具注册表 replace 选项

**位置**: [registry.ts:19-24](packages/core/src/tools/registry.ts)

**原问题**: `register()` 遇到同名工具直接抛异常，无法在运行时替换/升级工具实现。

**修复方案**: `register()` 增加可选参数 `options?: { replace?: boolean }`。默认行为不变（重复注册抛异常），传入 `{ replace: true }` 时允许覆盖。

```typescript
register(tool: Tool, options?: { replace?: boolean }): void {
    if (this.tools.has(tool.name)) {
        if (!options?.replace) {
            throw new Error(`工具 "${tool.name}" 已注册，不允许重复注册`);
        }
    }
    this.tools.set(tool.name, tool);
}
```

## 8. 上下文压缩阈值可配置化

**位置**: [loop.ts:751](packages/core/src/agent/loop.ts), [agent.ts:192-199](packages/core/src/types/agent.ts)

**原问题**: 压缩阈值硬编码为 85%，不同模型窗口差异大（Claude 200k vs GPT-3.5 4k），无法按需调整。

**修复方案**:
- `AgentConfig` 新增可选字段 `contextCompressThreshold?: number`（0-100，默认 85）
- `proactiveContextCheck()` 从 `config.contextCompressThreshold` 读取阈值

```typescript
const threshold = config.contextCompressThreshold ?? 85;
if (status.usagePercent >= threshold) { ... }
```
