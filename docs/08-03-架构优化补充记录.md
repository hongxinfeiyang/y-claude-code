## 1. 顶层可选字段迁移到 services

**原问题**: `AgentServices` 接口已定义但从未使用，33 处仍直接访问 `loopCtx.errorRecoveryManager` 等顶层字段，形成死代码。

**修复方案**: 所有 33 处顶层字段访问统一改为 `services` 优先 + 顶层兜底：

```typescript
// 修复前
loopCtx.observability?.startTurn(...)
loopCtx.errorRecoveryManager ? ... : null

// 修复后
(loopCtx.services?.observability ?? loopCtx.observability)?.startTurn(...)
(loopCtx.services?.errorRecovery ?? loopCtx.errorRecoveryManager) ? ... : null
```

迁移的 6 个字段：

| 字段 | 引用数 | 迁移后 |
|------|--------|--------|
| `errorRecoveryManager` | 9 | `services?.errorRecovery ?? errorRecoveryManager` |
| `observability` | 11 | `services?.observability ?? observability` |
| `contextMonitor` | 5 | `services?.contextMonitor ?? contextMonitor` |
| `summarizer` | 4 | `services?.summarizer ?? summarizer` |
| `cacheManager` | 1 | `services?.cacheManager ?? cacheManager` |
| `sandbox` | 1 | `services?.sandbox ?? sandbox` |

**注意**: `??` 优先级低于 `?.`，必须用括号包裹整个表达式后再链式调用。

## 2. 提取 LLMCallManager

**原问题**: `executeThinkingPhase()` 中的 LLM 流式调用、chunk 分发、token 追踪逻辑（~60 行 switch-case）内嵌在 AgentLoop 中。

**修复方案**: 提取 [llm-call-manager.ts](packages/core/src/agent/llm-call-manager.ts) (122 行)：

```typescript
class LLMCallManager {
    async *streamCall(
        messages: Message[],
        ctx: LLMCallContext,
        tokenUsage: TokenUsage,
    ): AsyncGenerator<TurnEvent, LLMCallResult>
}
```

**职责**:
- LLM 流式调用 (`provider.chat()`)
- chunk 分发 (text / thinking / tool_use / stop / error)
- token 使用量追踪
- 错误 chunk 分类（优先 `errorRecoveryManager.classifyErrorCode()`，不可用时回退内置 `classifyByCode()`）
- Cache / ContextMonitor 更新

**AgentLoop 调用简化为**:
```typescript
const result = yield* this.llmCallManager.streamCall(this.messages, {
    provider, model, signal, activeTools, ...
}, this.tokenUsage);
```

## 3. 提取 ToolExecutor

**原问题**: `executeToolPhase()` 中 ~140 行的工具执行管道（查找 → 熔断 → 权限 → 闸门 → 执行 → 错误处理）内嵌在 AgentLoop 中。

**修复方案**: 提取 [tool-executor.ts](packages/core/src/agent/tool-executor.ts) (163 行)：

```typescript
class ToolExecutor {
    async *executeAll(
        toolUses: ToolUse[],
        ctx: ToolExecutorContext,
        onStateChange: (state: string) => void,
        onConsecutiveError: (isError: boolean) => void,
    ): AsyncGenerator<TurnEvent, { toolResults: ToolResult[]; abort: boolean }>
}
```

**管道步骤**（按顺序）:
1. 查找工具 (`config.tools.find()`)
2. 熔断器检查 (`errorRecoveryManager.checkCircuitBreaker()`)
3. 权限检查 (`permissionManager.willPromptUser()` + `check()`)
4. 规划闸门 (`planModeManager.enforceGate()`)
5. 执行工具 (`tool.execute()`)
6. 错误处理 (`errorRecoveryManager.handleToolError()` + 熔断)

**回调机制**: AgentLoop 通过 `onStateChange` 和 `onConsecutiveError` 接收状态变更，保持内部状态 (`consecutiveErrors`, `state`) 同步。

**AgentLoop 调用简化为**:
```typescript
const result = yield* this.toolExecutor.executeAll(toolUses, {
    tools, permissionManager, errorRecoveryManager, planModeManager, ...
}, onStateChange, onConsecutiveError);
```
