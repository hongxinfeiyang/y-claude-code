## 1. AgentLoop 职责拆分

**原问题**: `AgentLoop` 单个类 (1035 行) 承载 11 项职责，违反单一职责原则 (SRP)。

**修复方案**: 提取 3 个新模块 + 1 个委托：

| 提取的模块 | 文件 | 职责 |
|-----------|------|------|
| `PlanModeManager` | [plan-mode-manager.ts](packages/core/src/agent/plan-mode-manager.ts) | 工具过滤、规划闸门、EnterPlanMode/ExitPlanMode 状态切换 |
| `ContextCompactor` | [context-compactor.ts](packages/core/src/agent/context-compactor.ts) | 消息压缩（Summarizer 语义摘要 + 简单截断降级） |
| `ErrorRecoveryManager.classifyErrorCode()` | [manager.ts](packages/core/src/agent/error-recovery/manager.ts) | 流式 error chunk 的错误码分类（原在 `AgentLoop.classifyChunkError`） |
| `MiddlewarePipeline` | [middleware.ts](packages/core/src/agent/middleware.ts) | 可插拔拦截器管道（LLM 前后、工具执行前后） |

**修复后 AgentLoop 职责** (~937 行，-98 行):

```
AgentLoop (6 项职责)
 ├─ ReAct 状态机编排          (核心)
 ├─ LLM 调用 + 流式处理       (executeThinkingPhase)
 ├─ 工具执行编排              (executeToolPhase)
 ├─ 轮次收尾                  (finalizeRound)
 ├─ 错误恢复决策              (handleLLMRecovery → ErrorRecoveryManager)
 └─ Token 截断                (truncateToolResults + estimateTokens)
```

**移除的方法**: `enforcePlanningGate` (42行), `handlePlanModeTransitions` (25行), `compactMessages` (32行), `classifyChunkError` (30行)

**移除的字段**: `planMode`, `originalTools`, `MODIFY_TOOLS`

## 2. AgentLoopContext 拆分为核心 + 服务

**原问题**: `AgentLoopContext` 12 个注入项平铺，8 个可选字段导致 27 处 optional chaining。

**修复方案**: 新增 `AgentServices` 子接口：

```typescript
// 核心依赖 (4 个必需)
export interface AgentLoopContext {
    permissionManager: PermissionManager;  // 必需
    sessionId: string;                     // 必需
    workingDirectory: string;              // 必需
    appendMessage: (content: string) => Promise<void>; // 必需
    signal?: AbortSignal;
    sandbox?: ISandbox;
    logger?: Logger;

    // 可选服务 — 收敛到子对象
    services?: AgentServices;

    // 向后兼容的顶层字段 (保留)
    errorRecoveryManager?: ErrorRecoveryManager;
    summarizer?: Summarizer;
    observability?: ObservabilityManager;
    cacheManager?: CacheManager;
    contextMonitor?: ContextMonitor;
}

export interface AgentServices {
    errorRecovery?: ErrorRecoveryManager;
    observability?: ObservabilityManager;
    summarizer?: Summarizer;
    cacheManager?: CacheManager;
    contextMonitor?: ContextMonitor;
    sandbox?: ISandbox;
}
```

**影响**: 新代码可通过 `loopCtx.services?.summarizer` 统一访问服务，保持向后兼容。

## 3. 错误恢复统一

**原问题**: 错误分类逻辑重复 — `AgentLoop.classifyChunkError` 和 `ErrorClassifier.classifyLLMError` 都在做 code→ErrorCategory 映射。

**修复方案**:
- `ErrorClassifier` 新增 `classifyByCode(code: string): ErrorCategory`（[classifier.ts](packages/core/src/agent/error-recovery/classifier.ts)）
- `ErrorRecoveryManager` 新增 `classifyErrorCode(code: string): ErrorCategory` 公开方法
- `AgentLoop` 移除 `classifyChunkError`，改为调用 `loopCtx.errorRecoveryManager?.classifyErrorCode(code)`

**决策流程统一**:
```
AgentLoop 流式 error chunk
  → ErrorRecoveryManager.classifyErrorCode(code)
    → ErrorClassifier.classifyByCode(code)
      → 单一 code→ErrorCategory 映射
```

## 4. 状态机显式转换

**原问题**: 状态转换分散在 14 处 `this.state = xxx`，部分通过 `return` 隐式退出循环。

**修复方案**: 新增 `setState()` 单入口方法：

```typescript
private setState(newState: AgentState, reason: string): void {
    if (this.state !== newState) {
        this.state = newState;
    }
}
```

所有 14 处状态转换统一通过 `setState(newState, "原因描述")` 调用，便于调试和审计。

## 5. Plan Mode 解耦

**原问题**: Plan Mode 的 3 层检查（工具过滤 / 闸门 / 状态切换）散布在 `AgentLoop` 的 3 个 phase 中，4 个专属字段 (`planMode`, `originalTools`, `planEnforcementMode`, `planState`)。

**修复方案**: 提取 `PlanModeManager` 类，暴露 3 个钩子：

| 钩子 | 调用位置 | 功能 |
|------|---------|------|
| `filterTools(tools)` | `executeThinkingPhase` 前 | 计划模式下过滤为只读工具 |
| `enforceGate(toolName)` | `executeToolPhase` 中 | 检查是否允许修改操作 |
| `handleTransitions(toolUses)` | `finalizeRound` 中 | 检测 EnterPlanMode/ExitPlanMode |

AgentLoop 中仅需：
```typescript
// Phase 1
const activeTools = this.planModeManager.filterTools(config.tools);

// Phase 2
const gateResult = this.planModeManager.enforceGate(toolUse.name);

// Phase 3
yield* this.planModeManager.handleTransitions(toolUses);
```

## 6. 权限检查 DRY 消除

**原问题**: `PermissionManager.check()` 和 `willPromptUser()` 约 60 行规则匹配代码完全重复。

**修复方案**: 新增 `decide()` 私有方法，统一缓存→规则→默认策略流程：

```typescript
private decide(toolUse: ToolUse, forceRecheck: boolean): PermissionDecision {
    // 1. Session 缓存检查
    // 2. 规则优先级排序 + pattern 匹配
    // 3. 默认策略回退
}
```

`check()` 和 `willPromptUser()` 均委托到 `decide()`：
- `check()`: `decide()` → allow→true, deny→false, prompt→promptUser()
- `willPromptUser()`: `decide()` → prompt→true, 其他→false

## 7. 中间件管道

**原问题**: 无拦截器机制，审计、限流、脱敏等横切关注点需要修改 `AgentLoop` 源码。

**修复方案**: 新增 `AgentMiddleware` 接口 + `MiddlewarePipeline` 执行器：

```typescript
export interface AgentMiddleware {
    readonly name: string;
    beforeLLMCall?(config: AgentConfig): Promise<AgentConfig>;
    afterLLMCall?(toolUses: ToolUse[]): Promise<ToolUse[]>;
    beforeToolExecution?(toolUse: ToolUse): Promise<{ blocked: boolean; reason?: string }>;
    afterToolExecution?(toolUse: ToolUse, result: ToolResult): Promise<ToolResult>;
    afterRound?(round: number, results: ToolResult[]): Promise<void>;
}
```

使用方式：
```typescript
const loop = new AgentLoop();
loop.middleware.use(auditMiddleware);      // 审计日志
loop.middleware.use(costControlMiddleware); // 成本控制
loop.middleware.use(desensitizeMiddleware); // 结果脱敏
```

管道按注册顺序执行，`beforeToolExecution` 支持短路阻止。
