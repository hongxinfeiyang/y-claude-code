## 📐 y-claude-code 架构设计分析

经过深入审视项目的核心架构文档和代码，我发现了**几个架构层面的问题**：

---

### 🔴 **1. Agent Loop 过度臃肿 - 职责混乱**

**问题**：
`AgentLoop` 类（~950 行）集合了太多不同层级的职责：

```
AgentLoop 的职责：
├─ ReAct 状态机编排          (核心)
├─ 工具调用顺序协调          (核心)
├─ LLM 调用管理              (应该抽离)
├─ 错误恢复决策              (应该抽离)
├─ 上下文压缩逻辑            (应该抽离)
├─ Permission 检查编排       (应该抽离)
├─ Plan Mode 状态管理        (应该抽离)
├─ Token 截断逻辑           (应该抽离)
└─ Observability 记录        (应该抽离)
```

**根本问题**：
- `executeThinkingPhase()` 处理 LLM 调用、错误分类、流式处理、错误恢复——这本该是 `LLMCallManager` 的职责
- `truncateToolResults()` 混入了 Token 估算、内容截断、日志记录——这本该是 `ToolResultProcessor` 的职责
- `compactMessages()` 混入了消息过滤、摘要生成、Summarizer 调用——这本该由独立的 `ContextCompactor` 处理

**当前架构**：
```
AgentLoop (950行)
 ├─ executeThinkingPhase (100行)    ← 做了 LLMCallManager 的事
 ├─ executeToolPhase (150行)        ← 做了 ToolExecutor 的事
 ├─ finalizeRound (50行)            ← 做了 RoundFinalizer 的事
 └─ [7个辅助方法，200行代码]
```

**建议的正确分层**：
```
AgentLoop (200行，只做编排)
 ├─ LLMCallManager            (处理 LLM 调用和流式响应)
 ├─ ToolPhaseExecutor         (处理工具执行编排)
 ├─ ContextCompactor          (处理上下文压缩逻辑)
 ├─ ToolResultProcessor       (处理结果截断和预处理)
 └─ PlanModeManager           (处理 Plan Mode 状态转换)
```

**违反的原则**：
- ❌ 单一职责原则（SRP）：AgentLoop 有 9+ 个改动原因
- ❌ 依赖反转原则（DIP）：直接调用 Summarizer、CacheManager 等，而非通过服务层
- ❌ 接口隔离原则（ISP）：AgentLoopContext 承载了 12 个注入依赖，很难扩展

### 🔴 **2. AgentLoopContext 依赖过多 - "神对象"反模式**

**问题**：
```typescript
export interface AgentLoopContext {
    permissionManager: PermissionManager;      // ✓ 必需
    sandbox?: ISandbox;                        // ✓ 必需
    logger?: Logger;                           // ✓ 必需
    sessionId: string;                         // ✓ 必需
    workingDirectory: string;                  // ✓ 必需
    appendMessage: callback;                   // ✓ 必需
    signal?: AbortSignal;                      // ✓ 必需
    errorRecoveryManager?: ErrorRecoveryManager;     // ⚠️ 可选，但关键
    summarizer?: Summarizer;                   // ⚠️ 可选，但关键
    observability?: ObservabilityManager;      // ⚠️ 可选
    cacheManager?: CacheManager;               // ⚠️ 可选
    contextMonitor?: ContextMonitor;           // ⚠️ 可选
}
```

**问题所在**：
- **12 个注入项**，却很难决定哪些是"核心"哪些是"可选"
- **11 个可选依赖**（标记 `?`），导致大量空值检查：
  ```typescript
  // 代码中充斥着这样的模式：
  if (loopCtx.errorRecoveryManager) { ... }
  loopCtx.observability?.recordLLMCall(...) ?? ""
  ```
- **依赖树不清晰**：不知道在何时应该注入什么，新增依赖时需要修改所有调用点

**后果**：
- 难以测试（需要 mock 12 个对象）
- 难以理解（不清楚哪些依赖是核心依赖）
- 难以扩展（新增功能时需要添加新的可选字段）

**建议**：
```typescript
// 分离为必需和可选两个接口
export interface AgentLoopContext {
    // 核心依赖 - 必需
    permissionManager: PermissionManager;
    sessionId: string;
    workingDirectory: string;
    signal: AbortSignal;
    logger: Logger;
    
    // 可选模块 - 通过服务容器注入
    services: AgentServices;
}

export interface AgentServices {
    errorRecovery?: ErrorRecoveryManager;
    observability?: ObservabilityManager;
    context?: ContextServices;  // 聚合 CacheManager + ContextMonitor + Summarizer
}

export interface ContextServices {
    cacheManager: CacheManager;
    monitor: ContextMonitor;
    summarizer: Summarizer;
}
```

### 🟡 **3. 错误恢复层级混乱 - 决策职责不清**

**问题**：错误恢复的决策逻辑散布在三个地方：

| 位置 | 决策 | 实现者 |
|------|------|--------|
| `executeThinkingPhase()` L391-472 | LLM 错误分类、流式 error 处理 | AgentLoop |
| `handleLLMRecovery()` L682-735 | LLM 恢复策略选择 | AgentLoop + ErrorRecoveryManager |
| `executeToolPhase()` L519-628 | 工具错误处理、熔断检查 | AgentLoop + ErrorRecoveryManager |

**具体问题**：

```typescript
// 问题 1: 重复的错误分类逻辑
if (chunk.type === "error") {
    errorCategory = this.classifyChunkError(chunk.code);  // AgentLoop 中
    const recovery = errorRecoveryManager.handleLLMError(err, ...); // ErrorRecoveryManager 中
}
// 两个地方都在做错误分类，不符合 DRY 原则

// 问题 2: 决策职责交织
// AgentLoop 决定"是否调用 ErrorRecoveryManager"
if (recovery.action === "retry" && recovery.success) { ... }
// 但 ErrorRecoveryManager 又被问"给我恢复策略"
// 谁最终决定"能否继续"？不清楚

// 问题 3: 熔断器被分开检查
// 工具执行前检查一次
if (loopCtx.errorRecoveryManager && !loopCtx.errorRecoveryManager.checkCircuitBreaker(toolUse.name)) { ... }
// 工具执行后又处理一次
if (recovery.strategy === RecoveryStrategy.CIRCUIT_BREAK) { return { abort: true, toolResults }; }
```

**现实影响**：
- 当添加新的恢复策略（如 rate limit 限流）时，需要修改三个地方
- 熔断器的"打开"和"检查"在不同位置，容易产生竞态条件
- 无法统一管理所有错误恢复的度量指标

**建议**：创建单一的错误决策中心

```typescript
// 创建 ErrorDecisionEngine
class ErrorDecisionEngine {
    async makeDecision(context: ErrorContext): Promise<ErrorDecision> {
        // 单一的决策点
        // 1. 分类错误
        // 2. 检查熔断器
        // 3. 选择恢复策略
        // 4. 返回统一的 ErrorDecision
    }
}

// AgentLoop 中简化为
const decision = await errorDecisionEngine.makeDecision({
    error, stage: 'llm_call', toolName, model
});

if (decision.action === 'retry') { /* 重试 */ }
else if (decision.action === 'switch_provider') { /* 切换 */ }
else if (decision.action === 'abort') { /* 终止 */ }
```

### 🟡 **4. 状态机实现不规范 - 隐式状态转换**

**问题**：状态机设计中存在"影子状态转换"：

```typescript
// 状态转换有两种方式：

// 方式1: 显式状态设置
this.state = AgentState.THINKING;
this.state = AgentState.EXECUTING;

// 方式2: 通过 return 隐式结束而不显式设置状态
if (thinkResult.exit) return;  // ← 状态没变，但循环退出了
```

**当前状态图有歧义**：
```
文档说: DONE → 应该是明确的结束状态
实际上:
  ├─ 显式 yield DONE 事件
  ├─ 隐式通过 return 退出循环（没有设置 state）
  └─ 有时 state 还停留在 THINKING 或 EXECUTING
```

**后果**：
- `getState()` 可能返回不准确的状态（用户中断时可能仍显示 THINKING）
- 基于 `state` 的条件判断可能失效
- 新开发者容易误解状态流转规则

**建议**：
```typescript
// 使用状态机库（如 xstate）或显式的 state 设置
export class AgentLoop {
    private setState(newState: AgentState, reason: string) {
        console.log(`[State Transition] ${this.state} → ${newState} (${reason})`);
        this.state = newState;
    }
    
    // 所有状态转换都显式调用
    this.setState(AgentState.THINKING, "LLM 推理开始");
    this.setState(AgentState.DONE, "用户中断");  // ← 确保 state 与实际行为对应
}
```

### 🟡 **5. Plan Mode 与主流程的耦合**

**问题**：Plan Mode 逻辑侵入 AgentLoop 的核心流程：

```typescript
// 在 executeThinkingPhase 中
const activeTools = this.planMode && this.originalTools.length > 0
    ? filterToolsForPlanMode(this.originalTools)        // ← Plan Mode 特殊逻辑
    : config.tools;

// 在 executeToolPhase 中
const gateResult = this.enforcePlanningGate(toolUse.name);  // ← Plan Mode 特殊逻辑
if (gateResult.blocked) { ... }

// 在 finalizeRound 中
yield* this.handlePlanModeTransitions(toolUses);  // ← Plan Mode 特殊逻辑
```

**问题分析**：
- Plan Mode 的三层检查散布在三个不同的阶段
- 规划强制执行（enforcePlanningGate）依赖 TodoWriteTool 的全局静态状态
- 无法清晰地禁用或替换 Plan Mode 行为

**建议**：提取 Plan Mode 为独立的"模式拦截器"

```typescript
// 创建 PlanModeInterceptor
interface AgentModeInterceptor {
    beforeLLMCall(config: AgentConfig): AgentConfig;      // 过滤工具
    beforeToolExecution(tool: ToolUse): InterceptionResult; // 检查是否阻止
    afterToolExecution(tool: ToolUse, result: ToolResult): void; // 追踪状态
}

class PlanModeInterceptor implements AgentModeInterceptor {
    beforeLLMCall(config: AgentConfig): AgentConfig {
        if (!this.planMode) return config;
        return {
            ...config,
            tools: filterToolsForPlanMode(config.tools)
        };
    }
    
    beforeToolExecution(tool: ToolUse): InterceptionResult {
        const gate = this.enforcePlanningGate(tool.name);
        return gate.blocked ? { blocked: true, reason: gate.message } : { blocked: false };
    }
}

// AgentLoop 中
for (const interceptor of this.interceptors) {
    const result = interceptor.beforeToolExecution(toolUse);
    if (result.blocked) { /* 拒绝 */ }
}
```

### 🟡 **6. 工具执行与权限检查的重复逻辑**

**问题**：Permission 检查在两个地方进行了冗余判断：

```typescript
// 1. executeToolPhase 中的检查
if (tool.requiresApproval(toolUse.input)) {
    if (loopCtx.permissionManager.willPromptUser(toolUse)) {
        this.state = AgentState.WAITING_APPROVAL;
        yield { type: "approval_request", tool: toolUse };
    }
    const approved = await loopCtx.permissionManager.check(toolUse);
    if (!approved) { /* 拒绝 */ }
}

// 2. PermissionManager 内部也做了类似的检查
async check(toolUse: ToolUse, forceRecheck = false): Promise<boolean> {
    const cacheKey = this.buildCacheKey(toolUse);
    if (!forceRecheck && this.sessionCache.has(cacheKey)) {
        return this.sessionCache.get(cacheKey) === "allow";
    }
    // 再次匹配规则...
}
```

**问题**：
- `willPromptUser()` 和 `check()` 的规则匹配逻辑重复（都做了 scope 排序、pattern 匹配）
- 两个方法的缓存检查不同步，可能导致行为不一致
- 规则匹配逻辑改动时需要在两处修改

**建议**：统一权限决策流程

```typescript
class PermissionDecision {
    action: 'allow' | 'deny' | 'prompt';
    reason: string;
    needsUI: boolean;
}

export class PermissionManager {
    // 单一的决策方法，同时返回决策和是否需要 UI
    async decide(toolUse: ToolUse): Promise<PermissionDecision> {
        // 一次规则匹配，返回完整决策
        const rule = this.findMatchingRule(toolUse);
        if (rule?.action === 'allow') return { action: 'allow', needsUI: false };
        if (rule?.action === 'ask') return { action: 'prompt', needsUI: true };
        // ...
    }
}

// AgentLoop 中简化为
const decision = await permissionManager.decide(toolUse);
if (decision.needsUI) {
    yield { type: 'approval_request', tool: toolUse };
}
if (decision.action === 'deny') { /* 拒绝 */ }
```

### 🟡 **7. 缺乏中间件机制 - 横切关注点难以管理**

**问题**：当前无法灵活地在 Agent 流程中插入自定义逻辑。

想象这样的场景：
- 审计：记录所有工具调用到审计日志
- 动态限流：根据时间段限制工具调用频率
- 自定义脱敏：在工具结果返回前脱敏敏感信息
- 成本控制：实时计算和限制 token 消耗

当前只能修改 AgentLoop 源码。文档承认了这一点：

> 缺少中间件机制：如果在 LLM 调用前后或工具执行前后需要插入自定义逻辑（审计、额外脱敏、自定义限流），当前只能修改 AgentLoop 源码。

**建议**：实现中间件管道

```typescript
export interface AgentMiddleware {
    beforeLLMCall?(config: AgentConfig): Promise<AgentConfig>;
    afterLLMCall?(result: ResponseChunk[]): Promise<ResponseChunk[]>;
    beforeToolExecution?(tool: ToolUse): Promise<ToolUse>;
    afterToolExecution?(tool: ToolUse, result: ToolResult): Promise<ToolResult>;
}

export class AgentLoop {
    private middlewares: AgentMiddleware[] = [];
    
    addMiddleware(mw: AgentMiddleware) {
        this.middlewares.push(mw);
    }
    
    private async executeMiddleware(
        event: 'beforeLLMCall' | 'afterToolExecution',
        data: any
    ): Promise<any> {
        let result = data;
        for (const mw of this.middlewares) {
            if (mw[event]) {
                result = await mw[event](result);
            }
        }
        return result;
    }
}

// 使用时
loop.addMiddleware(auditMiddleware);
loop.addMiddleware(costControlMiddleware);
loop.addMiddleware(deansitiveDataMiddleware);
```

### 🎯 **改进建议优先级**

**Phase 1（快速修复）**：
1. 将 Permission 检查逻辑从重复的两处统一
2. 显式设置所有状态转换，而不是隐式通过 return

**Phase 2（中期重构）**：
3. 从 AgentLoop 中提取 LLMCallManager 和 ToolPhaseExecutor
4. 分离 AgentLoopContext 为核心依赖和可选服务

**Phase 3（长期架构升级）**：
5. 实现中间件机制
6. 提取 PlanModeInterceptor 和 ErrorDecisionEngine
7. 使用状态机库规范化状态转换

这样的分阶段方案既能快速改进，又不会破坏现有功能。
