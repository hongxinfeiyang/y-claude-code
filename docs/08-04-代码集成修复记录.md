## N1 + N5: 中间件实际集成

**原问题**: `MiddlewarePipeline` 已完整实现 6 个钩子，但 `loop.ts` 和 `tool-executor.ts` 中完全没有调用。

**修复方案**: 在 4 个关键位置接入中间件：

| 位置 | 钩子 | 作用 |
|------|------|------|
| `executeThinkingPhase` 调用 LLM 前 | `beforeLLMCall` | 修改配置（注入工具、改 system prompt） |
| `executeThinkingPhase` LLM 返回后 | `afterLLMCall` | 过滤/修改 tool_use 列表 |
| `ToolExecutor.executeAll` 每个工具前 | `beforeToolExecution` | 阻止危险操作（短路机制） |
| `ToolExecutor.executeAll` 每个工具后 | `afterToolExecution` | 结果脱敏、审计记录 |
| `finalizeRound` 轮次结束 | `afterRound` | 统计、告警 |

`ToolExecutor.executeAll` 中 `beforeToolExecution` 返回 blocked 时，工具调用被跳过，错误信息注入 toolResults：

```typescript
if (ctx.middleware) {
    const mwResult = await ctx.middleware.runBeforeToolExecution(toolUse);
    if (mwResult.blocked) {
        toolResults.push({
            tool_use_id: toolUse.id,
            content: mwResult.reason ?? "操作被中间件阻止",
            is_error: true,
        });
        continue;
    }
}
```

## N3: 状态枚举

**原问题**: `onStateChange` 回调使用字符串 `"WAITING_APPROVAL"` / `"EXECUTING"`，loop.ts 中用 `if (s === "WAITING_APPROVAL")` 逐字比较。

**修复方案**: 新增 `ToolExecutorState` 枚举：

```typescript
export enum ToolExecutorState {
    WAITING_APPROVAL = "waiting_approval",
    EXECUTING = "executing",
}
```

回调签名: `onStateChange: (state: string) => void` → `onStateChange: (state: ToolExecutorState) => void`

调用方: `onStateChange("WAITING_APPROVAL")` → `onStateChange(ToolExecutorState.WAITING_APPROVAL)`

loop.ts 使用: `if (s === "WAITING_APPROVAL")` → `if (s === ToolExecutorState.WAITING_APPROVAL)`

## N4 + N8: DRY 消除

**原问题**: 22 处 `(loopCtx.services?.xxx ?? loopCtx.xxx)` 重复模式，ErrorRecoveryManager 等服务的解析逻辑分散各处。

**修复方案**: 新增 `svc()` 泛型辅助方法：

```typescript
private svc<T>(loopCtx: AgentLoopContext, key: keyof AgentServices): T | undefined {
    return (loopCtx.services?.[key] ?? loopCtx[key as keyof AgentLoopContext]) as T | undefined;
}
```

22 处内联模式全部替换为强类型调用：

```typescript
// 修复前
(loopCtx.services?.errorRecovery ?? loopCtx.errorRecoveryManager)
(loopCtx.services?.observability ?? loopCtx.observability)?.startTurn(...)

// 修复后
this.svc<ErrorRecoveryManager>(loopCtx, "errorRecovery")
this.svc<ObservabilityManager>(loopCtx, "observability")?.startTurn(...)
```

## N7: AbortSignal 增强

**原问题**: signal 只在 `run()` 主循环入口检查一次，子模块内无中断响应。用户在 LLM 流式调用或工具执行中途按 Ctrl+C 无法及时响应。

**修复方案**: 在 3 个位置新增中断检查：

| 位置 | 检查点 | 行为 |
|------|--------|------|
| `LLMCallManager.streamCall` | 每个 chunk 后 | `if (ctx.signal.aborted) break;` |
| `ToolExecutor.executeAll` | 每个 toolUse 循环前 | `if (ctx.signal.aborted) break;` |
| `loop.ts` `run()` 主循环 | 每轮开头 (已有) | setState(DONE), return |

signal 对象通过 Provider SDK 透传给底层 API（网络层也会响应中断）。
