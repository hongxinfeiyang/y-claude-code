## 2. 意图识别路由

**描述**: 所有用户输入在进入 LLM 之前，先经过 IntentRouter 进行意图分类。命中内置命令或明确工具调用模式的直接执行，避免不必要的 LLM 调用。

**三类路由结果**:

| 意图类型 | 识别方式 | 处理路径 | 延迟 |
|----------|----------|----------|------|
| 内置命令 | `/` 前缀 + 命令表匹配 | CommandHandler 直接执行 | < 1ms |
| 直接工具 | 中文/英文正则模式匹配 | ToolRegistry + 权限确认后执行 | < 1ms |
| 自然语言 | 以上均不命中（兜底） | 进入 Agent Loop → LLM | ~1-3s |

## 3. Agent 功能详设

### 3.1 多轮对话

**描述**: 用户与 AI 之间持续的多轮交互，AI 能记住历史上下文并自主调用工具。

**交互流程**:
```
1. 用户输入问题 (如 "帮我修复 src/utils.ts 中的 bug")
2. Agent 理解任务 → 调用 Read 读取文件 → 分析问题
3. Agent 提出修复方案 → 调用 Edit 修改代码
4. Agent 调用 Bash 运行测试 → 验证修复
5. Agent 汇报结果
```

**状态转换**: IDLE → THINKING → EXECUTING → THINKING → ... → DONE

**边界条件**:
- 用户中途按 Ctrl+C: 中断当前工具执行，允许重试或退出
- LLM 返回错误: 自动重试一次，仍失败则展示错误信息
- 工具执行超时: 默认 2 分钟超时，可配置

### 3.2 任务分解（Planner）

**描述**: 将复杂的用户请求自动拆分为可执行子任务，并用 TodoWrite 追踪进度。

**触发条件**: Agent 判断任务需要 3 个以上独立步骤时自动启用
**输出格式**:
```
📋 任务计划:
  1. [pending] 读取 src/utils.ts
  2. [pending] 定位 bug 根因
  3. [pending] 实现修复
  4. [pending] 运行测试验证
```

### 3.3 子代理（Sub-Agent）

**描述**: 通过 Agent 工具创建独立的子代理，处理可并行的子任务。

**使用场景**:
- 代码库探索: `Agent({ description: "Search for auth logic", prompt: "Find all authentication-related files" })`
- 独立审查: `Agent({ subagent_type: "code-reviewer", prompt: "Review migration 0042" })`
- 并行搜索: 同时启动多个 Explore agent

**约束**:
- 子代理有独立的上下文窗口，不污染主会话
- 子代理的结果以摘要形式返回主代理
- 默认超时 5 分钟

### 3.6 Plan Mode（计划模式）

**描述**: 在执行复杂任务前，Agent 可先进入受控的"计划模式"，只用只读工具调研代码，生成实现计划供用户审批。

**触发方式**:
- Agent 自动判断（任务复杂度高时）
- 用户显式要求："请先制定计划，不要直接修改代码"

**工作流程**:
```
1. Agent 调用 EnterPlanMode → 进入 PLANNING 状态
2. 工具过滤：仅允许 Read / Glob / Grep 等只读工具
3. Agent 调研代码 → 制定实现计划
4. Agent 调用 ExitPlanMode → 提交计划给用户
5. 用户在终端中审核 Markdown 格式的计划
6. 用户批准 → Agent 按计划逐步实现
7. 用户拒绝 → Agent 调整方案并重新提交
```

**安全约束**:
- 计划模式下所有写操作工具（Write/Edit/Bash）被过滤
- ExitPlanMode 需要用户显式审批（y/n 交互确认）
- 计划内容以 Markdown 渲染展示在终端中

### 3.7 思考模式（Extended Thinking）

**描述**: 利用 Anthropic Claude 的 extended thinking 能力，让模型在复杂推理前进行深度思考。

**配置**:
- `thinkingEnabled: true` 开启
- `thinkingTokens: 4000` 思考预算（默认）
- 思考内容以折叠方式展示给用户

### 3.5 错误恢复

**策略表**:
| 错误类型 | 处理方式 |
|----------|----------|
| LLM API 超时 | 自动重试 1 次，指数退避 |
| LLM 返回格式异常 | 重新发送请求，附错误提示 |
| 工具执行失败 | 将错误信息注入上下文，Agent 自行调整 |
| Token 超限 | 压缩历史 → 重试 |
| 网络错误 | 3 次重试，每次 2x 退避 |

## 4. 工具功能详设

### 4.1 Read — 文件读取

| 属性 | 值 |
|------|-----|
| 权限 | readonly |
| 支持格式 | .txt, .md, .js, .ts, .json, .py, .go, 等所有文本文件；.png/.jpg 图片（多模态） |
| 行号 | 自动添加行号前缀，格式: `行号\t内容` |
| 截断 | 默认 2000 行，超长文件提示使用 offset/limit |
| PDF | 支持读取，指定 pages 参数 |

### 4.2 Write — 文件写入

| 属性 | 值 |
|------|-----|
| 权限 | write |
| 安全约束 | 路径限制在工作目录内，拒绝 `../` 越权 |
| 覆盖策略 | 默认覆盖已存在文件，创建不存在的目录 |
| 编码 | 默认 UTF-8 |

### 4.3 Edit — 精确字符串替换

| 属性 | 值 |
|------|-----|
| 权限 | write |
| 核心算法 | 精确字符串匹配 → 替换 → 写回 |
| 唯一性检查 | old_string 必须在文件中唯一（除非 replace_all） |
| 非唯一报错 | 返回错误，列出所有匹配位置及上下文 |
| replace_all | 替换所有匹配项 |

### 4.4 Bash — Shell 执行

| 属性 | 值 |
|------|-----|
| 权限 | exec |
| 超时 | 默认 120s，可配置 (max 600s) |

| 后台执行 | `run_in_background: true` 支持后台运行 |

### 4.5 Glob — 文件搜索

| 属性 | 值 |

### 4.6 Grep — 内容搜索

### 5.2 权限确认界面

```
┌──────────────────────────────────────────────────┐
│  🔐 需要执行命令执行: npm install react，是否允许？│
│                                                   │
│  (y=允许 / n=拒绝 / a=本会话全允许此类操作):      │
│                                                   │
│  y → ✓ 已允许（后续相同操作自动放行）              │
│  a → ✓ 已允许（本次会话自动放行所有 命令执行）     │
│  n → ✗ 已拒绝                                    │
└──────────────────────────────────────────────────┘
```

**自动记忆机制**:
- 按 `y`：PermissionManager 通过 `buildCacheKey` 生成细粒度 key
  - Bash: `Bash:npm`（按命令基名，允许 npm 不会连带允许 rm）
  - Write/Edit: `Write:/path/to/file`（按文件路径，允许写 fileA 不会连带允许 fileB）
  - 其他工具: 工具名
- 按 `a`：以工具名为宽泛 key 写入 session 缓存（本会话全允许此类操作）
- 记忆仅在当前 session 有效，关闭终端后失效
- `willPromptUser()` 预检：同步判断是否需要弹窗，用于避免不必要的"等待确认"提示

### 5.3 流式输出渲染

**StreamingMarkdownRenderer**: 将 LLM 流式文本中的 Markdown 语法转为 ANSI 终端格式。

| 功能 | 实现 |
|------|------|
| 代码块语法高亮 | ANSI 颜色标记关键字/字符串/注释/数字 |
| 行内格式 | `**粗体**` → ANSI bold，`` `code` `` → 黄色，`# 标题` → bold |
| 跨 chunk 行缓冲 | 不完整行保留在 `lineBuffer`，完整行才消费渲染 |
| 空白行压缩 | 跨 chunk 追踪连续空白行数（`blankRun`），最多保留 2 个连续空行，超出静默丢弃 |
| 缓冲区冲刷 | `flush()` 处理未闭合代码块和残余半行 |

**工具调用展示增强**: 每个工具调用展示关键参数（60 字符截断）

```
⚙ 读取文件 src/app.ts 执行中...
⚙ 搜索文件 **/*.ts 执行中...
⚙ 执行命令 npm test 执行中...
⚙ 获取网页 https://example.com 执行中...
⚙ 网络搜索 your-query 执行中...
⚙ 启动子代理 Search for auth 执行中...
```

**spinner 集成**:
- 120ms 间隔旋转指示器（⠋⠙⠹⠸⠼⠴⠦⠧⠇⠏）
- 首次文本输出时自动停止（`textStarted` 标志）
- 所有事件输出前通过 `writeLine()` 清除 spinner 行（`readline.clearLine` + `cursorTo`，避免裸 `\r` 残留）

### 6.3 LLM 自助配置

**描述**: 用户无需手动编辑配置文件，通过终端交互界面 (`--setup`) 即可完成 LLM 提供商的配置。

**配置流程**:

```text
首次启动 → 自动检测无配置 → 弹出配置向导
  1. 选择 Provider (Anthropic / OpenAI / 自定义)
  2. 输入 API Key（脱敏显示）
  3. 可选输入自定义 Base URL
  4. 选择默认模型
  5. 测试连接 → 通过则保存
```

**配置项**:

| 项 | 说明 | 必填 |
| --- | --- | --- |
| Provider | 选择 Anthropic / OpenAI / 自定义 | 是 |
| API Key | 对应提供商的 API 密钥 | 是 |
| Base URL | 自定义 API 端点（代理/私有部署） | 否 |
| 默认模型 | 如 `claude-sonnet-4-6` / `gpt-4o` | 是 |

**Provider 管理**:

- 支持同时配置多个 Provider
- 运行时通过 `/model` 或配置面板随时切换
- API Key 存储于 `~/.y-claude-code/config.json`，权限 600
- 支持从环境变量读取（优先于配置文件）

**重新配置入口**:

- `y-claude-code --setup` — 进入配置向导
- `/config provider` — 命令方式切换 Provider
- `/model <name>` — 命令方式切换模型

## 7. Hook 功能详设

### 7.1 事件类型

| 事件 | 触发时机 | 参数 |
|------|----------|------|
| `before:tool:execute` | 工具执行前 | toolName, params, context |

```json
        ]
    }
}
```

## 8. Memory 功能详设

### 8.1 Memory 类型

| 类型 | 用途 | 存储位置 |
|------|------|----------|
| user | 用户角色、偏好、知识背景 | `~/.y-claude-code/memory/` |
| feedback | 用户反馈的行为指导 | `~/.y-claude-code/memory/` |

| reference | 外部系统引用指针 | `~/.y-claude-code/memory/` |

### 8.2 Memory 文件格式

```markdown
---
name: user-preference
description: 用户偏好 TypeScript strict mode
metadata:
  type: user
---

用户在所有项目中偏好使用 TypeScript strict mode。
**Why:** 类型安全能减少运行时错误。
**How to apply:** 生成代码时默认开启 strict: true。
```

## 10. Cron 定时任务详设

### 10.1 功能描述

支持定时触发 Agent 执行指定任务，如定期检查部署状态、定时运行测试。

### 10.2 配置

```json
{
    "cron": "*/5 * * * *",
    "prompt": "检查部署状态",
    "recurring": true,
    "durable": false
}
```

### 10.3 约束

- 最小间隔 60 秒
- 仅在 Agent 空闲时触发
- 持久任务写入 `.claude/scheduled_tasks.json`
- 重复任务 7 天后自动过期

## 11. Docker 沙箱功能详设

### 11.1 功能描述

所有 Bash 工具执行的命令运行在独立的 Docker 容器中，与宿主机环境隔离。沙箱为独立包 `@y-claude-code/sandbox`，通过接口注入到 core。

### 11.2 容器规格

| 属性 | 默认值 | 说明 |
| --- | --- | --- |
| 基础镜像 | `node:20-alpine` | 首次启动自动拉取 |
| 工作目录 | `/workspace` | 挂载项目根目录 |
| 网络模式 | `none`（默认断网） | Bash 工具显式声明才开启 bridge |
| 内存限制 | 256MB | 可配置，最大 1GB |
| CPU 限制 | 不限 | 可配置为 1.0（单核） |
| 超时 | 120s | 超时后 SIGKILL 强杀 |
| stdout 截断 | 1MB | 超出截断并标注 |
| 自动清理 | 是 | 容器执行完自动删除 |

### 11.3 安全策略

| 规则 | 说明 |
| --- | --- |
| 危险命令拦截 | `rm -rf /`、`mkfs.`、`dd`、fork bomb 等正则匹配拒绝执行 |
| 网络隔离 | 默认关闭网络，防止反弹 Shell、数据外泄 |
| 挂载白名单 | 仅允许挂载项目工作目录，拒绝 `/etc`、`/root`、`/proc` 等敏感路径 |
| 资源限制 | 内存/CPU 上限防止 DoS |
| 用户隔离 | 容器内以非 root 用户运行（uid 1000） |

### 11.4 不可用时的降级策略

```text
Docker 不可用 → 提示用户安装 Docker 或选择降级为本地执行
  - 本地执行: 安全策略弱化为仅做命令正则校验 + 路径校验
  - 警告色显示 "⚠ 未启用沙箱"
```

## 12. 会话管理功能详设

### 12.1 功能描述

SessionManager 管理会话的完整生命周期：创建、恢复、列表、删除、消息持久化。

### 12.2 核心功能

| 功能 | 说明 |
| --- | --- |
| 自动保存 | 每条消息追加后自动写入 `~/.y-claude-code/sessions/{id}.json` |
| 会话恢复 | `y-claude-code --resume <id>` 恢复中断的会话 |
| 会话列表 | `y-claude-code --sessions` 列出历史会话及预览 |
| 会话删除 | `y-claude-code --delete <id>` 删除指定会话 |
| 自动命名 | 基于首条用户消息生成会话预览文本 |
| 跨会话上下文 | 可选：新会话可导入旧会话的摘要 |
