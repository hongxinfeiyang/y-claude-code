# 01 — 系统架构

## 1. 项目概述

本项目是一个类 Claude Code 的 AI 编程助手，通过 LLM Agent 模式辅助开发者完成编码任务。产品形态包含 CLI 终端工具和 VS Code 插件，支持 Anthropic Claude 与 OpenAI GPT 两个 LLM 后端。

### 2.1 接入层

| 组件 | 技术 | 职责 |
|------|------|------|
| CLI | Node.js readline | 终端交互、流式输出、Markdown 渲染（ANSI 代码高亮+空白行压缩）、权限确认、LLM 自助配置、沙箱注入 |
| VS Code Extension | VS Code Extension API | IDE 内联补全、Webview 面板（工具审批含 Accept/Reject 按钮）、语言服务 |
| HTTP API (预留) | Fastify | 为 Web 端或第三方集成提供 REST 接口 |

**依赖注入**: CLI 层启动时创建 `DockerSandboxManager` 实例，通过 `AgentConfig` 注入到 core 引擎。core 包仅依赖 `ISandbox` 接口，不感知 Docker 实现。

### 2.2 核心引擎层

核心引擎包 `@y-claude-code/core` 是所有上层接入的共享运行时，以库的形式被 CLI / VS Code 插件引用。

核心设计原则：**core 包零外部副作用依赖**。Docker（dockerode）、文件系统等由接入层注入接口实现。

| 模块 | 职责 |
|------|------|
| IntentRouter | 用户输入意图识别：内置命令 / 直接工具调用 / 自然语言 |
| Agent Loop | ReAct 循环：推理 → 工具调用 → 观察 → 再推理 |
| Plan Mode | 计划模式：EnterPlanMode/ExitPlanMode + PLANNING 状态 + 只读工具过滤 |
| ErrorRecovery | 错误分类、指数退避重试、Provider 回退链、熔断器、上下文溢出恢复 |
| 工具系统 | 20 个内置工具 + MCP 适配 + 自定义注册 |
| 上下文管理 | System prompt / Token 预算 / 渐进式 LLM 摘要 (Summarizer) / Cache 管理 / 窗口监控 |
| 权限系统 | 分级权限 (read/write/exec/network) + 规则引擎 + 交互确认 |
| Hook 系统 | 生命周期事件钩子 (before/after:tool/llm, session, error) |
| Memory 系统 | 用户/项目记忆持久化（user/feedback/project/reference 四类） |
| Session 管理 | 会话创建/恢复、消息历史持久化、跨会话状态 |
| Skill 系统 | Markdown 技能定义 + frontmatter + 三级优先级（内置/项目/用户） |
| Cron 调度 | 定时任务（一次性 + 重复），持久化到 `.claude/scheduled_tasks.json` |
| 安全工具 | Prompt Injection 检测(10 种模式)、输出脱敏(9 类)、SSRF 防护、命令注入检测 |
| 自动更新 | AutoUpdateManager：npm registry 检查 + 24h 节流 + 启动时异步通知 |
| 遥测 | TelemetryManager：匿名安装 ID + 本地 JSONL 存储 + 环境变量/配置开关 |
| tmux 集成 | TmuxManager：环境检测 + 窗口/面板管理 + 内容捕获 |
| 可观测性 | JSONL Transcript、Metrics(Counter/Histogram/Gauge)、Span Tracing |
| LLM 适配 | Provider 抽象 + Anthropic/OpenAI 实现 + Provider 回退链 |

### 2.3 基础设施层

| 组件 | 职责 |
|------|------|
| 日志 | 结构化日志，分级输出（debug/info/warn/error） |
| 存储 | 会话记录、配置、Memory 文件、Skill 定义的持久化 |
| MCP | 实现 Model Context Protocol，支持外部工具和资源接入 |
| Cron | 定时任务调度，支持一次性/重复任务，持久化到文件 |

### 2.4 独立服务层 — Docker 沙箱

沙箱从基础设施层独立出来，作为独立包 `@y-claude-code/sandbox`，原因：

- **独立依赖**: 沙箱依赖 `dockerode`（Docker Engine API 客户端），不应污染 core 包
- **多消费者**: Bash 工具、Python 执行器、子代理隔离、worktree 隔离均可复用
- **接口解耦**: core 只依赖 `ISandbox` 接口，沙箱实现由 CLI 在运行时注入
- **可替换**: 用户可选择 Docker / Podman / 本地执行（无需沙箱）

### 3.2 LLM 请求/响应数据流

```
Agent Loop
  │
  ├──> LLMProvider.chat(messages, tools, options)
  │       │
  │       ├── AnthropicProvider → @anthropic-ai/sdk
  │       │       └── POST https://api.anthropic.com/v1/messages
  │       │
  │       └── OpenAIProvider → openai SDK
  │               └── POST https://api.openai.com/v1/chat/completions
  │
  └──> AsyncIterable<ResponseChunk>
          ├── { type: "text", content: "..." }
          ├── { type: "tool_use", name: "read", input: {...} }
          ├── { type: "thinking", content: "..." }
          └── { type: "stop", reason: "end_turn" }
```

## 6. 技术选型依据

| 类别 | 选型 | 核心考量 |
|------|------|----------|
| 语言 | TypeScript 5.x strict | 类型安全防止 Agent 流程中消息格式错误 |
| 运行时 | Node.js 20+ | LTS 稳定，流式处理和 AsyncGenerator 原生支持 |
| 包管理 | pnpm + workspace | 高效磁盘使用，严格依赖隔离 |
| CLI 框架 | Node.js readline | 终端原生交互，零依赖，历史记录内建支持 |
| LLM SDK | @anthropic-ai/sdk + openai | 官方维护，紧跟 API 更新 |
| Token 计数 | tiktoken (wasm) | OpenAI tokenizer 标准；Anthropic 侧用近似估计 |
| 配置 | cosmiconfig | 自动发现 `.claude/settings.json` / `y-claude-code.config.js` / `package.json` 中的配置 |
| 沙箱 | Docker | 容器级隔离执行 Bash 命令，限制网络和文件系统 |
| 测试 | vitest | 快速、兼容 Jest API、原生 ESM |
| 构建 | tsup | esbuild 打包，速度快，支持多入口 |

## 7. 并发与性能模型

- **单 Agent 实例**: 每个会话对应一个 Agent Loop 实例，单线程运行
- **子代理**: 通过 `Agent` 工具创建独立子代理，可并发执行（Promise.all）
- **流式输出**: 所有 LLM 调用走 stream，避免用户长时间等待
- **Prompt Cache**: Anthropic 侧利用 prompt caching，将 system prompt 和 tools 定义标记为可缓存，显著降低延迟和成本
- **Token 预计算**: 在构建上下文时实时累加 token 数，避免超限后再截断

## 10. 错误恢复模型

### 10.1 设计动机

当前 Agent Loop 的错误处理存在以下不足：

| 问题 | 现状 | 影响 |
|------|------|------|
| LLM 调用错误直接终止 | 网络超时 / 429 限流 / 5xx 服务端错误一律 `return` | 一次波动即中断用户工作流 |
| 工具错误无区分 | 未知工具、用户拒绝、执行异常全部计入 `consecutiveErrors` | 性质不同的错误被等同对待 |
| 无重试机制 | Provider 层和 Agent Loop 层均无重试 | 瞬时故障无法自愈 |
| 无熔断保护 | 只有 `consecutiveErrors >= 3` 硬终止 | 无法区分间歇性故障和系统性故障 |
| 无 Provider 回退 | 主 Provider 故障后无自动切换 | 单点依赖，可用性低 |
| 错误信息不结构化 | 仅 `Error.message` 字符串 | Hook / 日志系统无法按错误类型分流处理 |

### 10.2 错误分类体系

```
                          ┌─────────────────┐
                          │   Agent 错误     │
                          └────────┬────────┘
                                   │
              ┌────────────────────┼────────────────────┐
              │                    │                    │
     ┌────────▼────────┐  ┌───────▼───────┐  ┌────────▼────────┐
     │  LLM 层错误      │  │  工具层错误    │  │  系统层错误      │
     │ (Provider 侧)    │  │ (Tool 侧)      │  │ (Loop 内部)      │
     └────────┬────────┘  └───────┬───────┘  └────────┬────────┘
              │                    │                    │
     ┌────────┼────────┐  ┌───────┼───────┐  ┌────────┼────────┐
     │        │        │  │       │       │  │        │        │
  NETWORK  RATE_   AUTH  TOOL_   TOOL_  PERM_  CONTEXT  MAX_
           LIMIT         NOT_    EXEC   DENIED OVERFLOW ROUNDS
                         FOUND
```

**错误类别枚举与恢复策略映射**:

| 错误类别 | 来源层 | 可重试 | 恢复策略 | 说明 |
|----------|--------|--------|----------|------|
| `NETWORK` | LLM | 是 | 指数退避重试 (最多 3 次) | DNS 解析失败、连接超时、TCP 断开 |
| `RATE_LIMIT` | LLM | 是 | 等 Retry-After + 退避重试 | HTTP 429，尊重服务端 `Retry-After` 头 |
| `PROVIDER_ERROR` | LLM | 是 | 退避重试 1 次 → 切换 Provider | HTTP 5xx、服务暂不可用 |
| `AUTH` | LLM | 否 | 立即终止 + 提示配置 API Key | HTTP 401/403，无法自动恢复 |
| `INVALID_REQUEST` | LLM | 否 | 立即终止 + 展示请求详情 | HTTP 400，参数格式错误（不应发生） |
| `CONTEXT_OVERFLOW` | LLM | 否 | 自动触发上下文压缩后重试 1 次 | 上下文超限，压缩后重试 |
| `TOOL_NOT_FOUND` | 工具 | 否 | 向 LLM 反馈未知工具 → 继续循环 | LLM 幻觉产生的工具名 |
| `TOOL_EXEC` | 工具 | 部分 | 反馈错误信息给 LLM → 继续循环 | 工具执行异常（文件不存在等） |
| `PERM_DENIED` | 工具 | 否 | 反馈拒绝信息给 LLM → 继续循环 | 用户拒绝授权 |
| `MAX_ROUNDS` | 系统 | 否 | 强制要求 LLM 总结后终止 | 达到 maxToolRounds |
| `CIRCUIT_BREAKER` | 系统 | 否 | 熔断打开 → 终止会话 | 同一工具连续失败触发 |

### 10.3 恢复策略定义

```
┌─────────────────────────────────────────────────────────────────┐
│                     ErrorRecoveryManager                         │
│                                                                  │
│  classifyError(error) → ErrorCategory                           │
│  getStrategy(category, attemptCount) → RecoveryAction           │
│  executeStrategy(action, context) → RecoveryResult              │
│                                                                  │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────────────┐  │
│  │ RetryManager │  │CircuitBreaker│  │ ProviderFailoverMgr  │  │
│  │              │  │              │  │                      │  │
│  │ maxRetries:3 │  │ threshold:5  │  │ providers: [primary, │  │
│  │ baseDelay:1s │  │ windowMs:60s │  │             fallback]│  │
│  │ maxDelay:30s │  │ halfOpenMs:  │  │ currentIndex: 0      │  │
│  │ multiplier:2 │  │          30s │  │                      │  │
│  └──────────────┘  └──────────────┘  └──────────────────────┘  │
└─────────────────────────────────────────────────────────────────┘
```

**五种核心策略**:

| 策略 | 触发条件 | 行为 | 参数 |
|------|----------|------|------|
| `RETRY` | NETWORK / RATE_LIMIT / PROVIDER_ERROR | 等待后退时间后重新发起相同 LLM 请求 | 基础延迟 1s，最大延迟 30s，退避乘数 2 |
| `SWITCH_PROVIDER` | PROVIDER_ERROR 重试仍失败 | 切换到 `fallbackProvider`，重置 attempt 计数 | 回退链: Anthropic → OpenAI |
| `COMPACT_CONTEXT` | CONTEXT_OVERFLOW | 触发 LLM 驱动上下文压缩，削去最早 50% 历史 | 压缩后重试 1 次 |
| `FEEDBACK_TO_LLM` | TOOL_EXEC / TOOL_NOT_FOUND / PERM_DENIED | 将错误信息以 `is_error: true` 的 tool_result 注入上下文 | 不终止循环，LLM 自行调整 |
| `CIRCUIT_BREAK` | 同一工具 60s 内连续失败 5 次 | 熔断打开，该工具 30s 内拒绝执行，直接返回错误 | halfOpenMs 后可放行一次探测调用 |

### 10.4 重试退避算法

```
重试间隔 = min(baseDelay × multiplier^(attempt-1) + jitter, maxDelay)

其中:
  baseDelay   = 1000ms（初始等待时间）
  multiplier  = 2（指数退避因子）
  maxDelay    = 30000ms（最大等待上限）
  jitter      = random(0, baseDelay × multiplier^(attempt-1) × 0.1)
              （随机抖动 0~10%，避免惊群效应）

重试序列: 1s → 2s → 4s → 8s → 16s → 30s (上限)
```

**特殊处理 — RATE_LIMIT**: 若 LLM 返回 `Retry-After` 头，则以 `Retry-After` 值作为最小等待时间：
```
实际等待 = max(Retry-After秒, 退避计算结果)
```

### 10.7 Provider 回退链

```
用户配置的 Provider 优先级:

  primary: "anthropic" ──→ fallback: "openai"
       │                          │
       │ 故障                      │ 故障
       ▼                          ▼
  切换至 openai              终止 + 提示用户
  (保留原 model 映射)         "所有 Provider 不可用"
```

**模型映射规则**: 每个 Provider 维护一个模型名映射表，切换 Provider 时自动选择能力最接近的模型。

## 11. 上下文管理与渐进式摘要

### 11.1 设计动机

当前上下文压缩为纯文本截断：`compressHistory()` 将旧消息截取前 200 字符拼接，`compactMessages()` 直接丢弃 50% 消息只留占位符。这在长对话场景中导致关键信息丢失，Agent 行为退化。

**升级目标**: 用 LLM 驱动的渐进式摘要替换文本截断，保留语义信息的同时回收 Token。

### 11.2 渐进式摘要架构

```
ContextBuilder.build()
      │
      ▼
┌─────────────────────────────┐
│ TokenBudgetManager          │
│  检查 messages token 数      │

└──────────┬──────────────────┘
           │ 超过阈值
           ▼
┌─────────────────────────────┐
│ Summarizer                  │
│                              │
│  ┌────────────────────────┐ │
│  │ 1. 消息分块             │ │
│  │    ┌──────┐ ┌──────┐   │ │
│  │    │Chunk1│ │Chunk2│   │ │
│  │    │(20条)│ │(20条)│   │ │
│  │    └──┬───┘ └──┬───┘   │ │
│  │       │        │       │ │

│  │  │ 2. LLM 逐块摘要   │  │ │
```

### 11.3 摘要策略配置

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `summarizeThreshold` | 0.70 | 消息 token 占比超过此值触发摘要 |
| `chunkSize` | 20 | 每次 LLM 摘要调用处理的消息条数 |
| `recentPreserveCount` | 10 | 始终保留最近的 N 条消息不被摘要 |
| `summaryMaxTokens` | 2000 | 单次摘要调用的输出 token 上限 |
| `maxSummaryTokens` | 8000 | 摘要文本累积的最大 token 数 |
| `progressiveMerge` | true | 是否将新旧摘要合并（渐进式） |

### 11.4 摘要专用 Prompt

```markdown

规则:
1. 保留所有关键决策、代码变更、错误信息和用户指令
2. 保留文件路径、函数名、技术术语等精确信息
3. 省略闲聊、重复确认和过程性描述
4. 用中文输出，每段不超过 3 句话
5. 格式: "## 摘要\n- 关键点1\n- 关键点2\n..."
```

### 11.5 摘要缓存与合并

```
首次摘要:
  旧消息(20条) → LLM摘要 → Summary_v1

增量合并 (progressiveMerge = true):
  Summary_v1 + 新消息(20条) → LLM → Summary_v2
  (摘要 + 新消息一起送入 LLM，输出合并后的新摘要)

摘要 Token 超限处理:
  若 Summary_vN 超过 maxSummaryTokens:
    Summary_vN → LLM → Summary_compressed (自摘要)
```

### 11.6 与现有系统的集成

| 集成点 | 改动 |
|--------|------|
| `ContextBuilder.compressHistory()` | 注入 Summarizer，优先走 LLM 摘要路径，失败回退文本截断 |
| `AgentLoop.compactMessages()` | 替换占位符为 Summarizer 产出的真实摘要 |
| `AgentConfig` | 新增 `summarizeConfig` 配置项 |
| `AgentLoopContext` | 新增 `summarizer` 可选依赖 |

```jsonc
{
  "agent": {
    "summarize": {
      "enabled": true,
      "threshold": 0.70,
      "chunkSize": 20,
      "recentPreserveCount": 10,
      "summaryMaxTokens": 2000,
      "maxSummaryTokens": 8000,
      "progressiveMerge": true
    }
  }
}
```

## 12. 可观测性

### 12.1 设计动机

当前系统只有基础的结构化日志（`Logger`），缺少：
- **对话回放**: 无法回溯 Agent 的完整决策过程
- **性能度量**: 无法量化 LLM 延迟、工具耗时、Token 消耗趋势
- **错误聚合**: 无法按错误类型统计发生率
- **链路追踪**: 无法关联"用户输入 → LLM 调用 → 工具执行 → 下一轮 LLM"的因果链

### 12.2 三维可观测性架构

```
┌─────────────────────────────────────────────────────────────┐
│                   ObservabilityManager                        │
│                                                              │
│  ┌──────────────────┐  ┌──────────────┐  ┌───────────────┐ │
│  │  Transcript      │  │  Metrics      │  │  Tracer       │ │
│  │  (JSONL 记录)    │  │  (指标收集)   │  │  (链路追踪)   │ │
│  │                  │  │              │  │               │ │
│  │  每轮 TurnEvent  │  │  Counter     │  │  Trace ID     │ │
│  │  工具调用/结果   │  │  Histogram   │  │  Span ID      │ │
│  │  错误信息        │  │  Gauge       │  │  父子关系     │ │
│  │  Done 统计       │  │              │  │               │ │
│  └────────┬─────────┘  └──────┬───────┘  └───────┬───────┘ │
│           │                   │                   │         │
│           ▼                   ▼                   ▼         │
│     ~/.y-claude-code/   内存中聚合，按需      内存中维护     │
│    transcripts/         导出为 JSON            Span 树       │
│    session-{id}.jsonl                                  │
└─────────────────────────────────────────────────────────────┘
```

### 12.3 Transcript（JSONL 对话记录）

**格式**: 每行一个 JSON 对象，追加写入 `~/.y-claude-code/transcripts/{sessionId}.jsonl`

**记录事件类型**:

| 事件 | 记录内容 | 触发时机 |
|------|----------|----------|
| `turn:start` | turn 序号、时间戳、用户输入（截断至 500 字符） | AgentLoop.run() 开始时 |
| `llm:call` | 模型名、消息数、预估 token 数、时间戳 | LLM 调用开始时 |
| `llm:result` | 延迟 ms、输入/输出 token、tool_use 数量、是否错误 | LLM 调用完成时 |
| `tool:call` | 工具名、参数 keys（不含值）、时间戳 | 工具调用开始时 |
| `tool:result` | 延迟 ms、是否错误、结果长度 | 工具调用完成时 |
| `error` | 错误类别、消息、恢复策略 | 发生错误时 |
| `recovery` | 恢复策略、成功/失败、延迟 | 错误恢复时 |
| `turn:end` | 总耗时、总 token、工具调用次数、轮次 | AgentLoop.run() 结束时 |

### 12.4 Metrics（性能指标）

**计数器 (Counter)**:

| 指标名 | 说明 |
|--------|------|
| `llm.calls.total` | LLM 调用总次数 |
| `llm.calls.errors` | LLM 调用错误次数（按类别分组） |
| `tool.calls.total` | 工具调用总次数 |
| `tool.calls.errors` | 工具调用错误次数（按工具名分组） |
| `recovery.attempts` | 恢复尝试次数（按策略分组） |
| `recovery.successes` | 恢复成功次数 |

**直方图 (Histogram)**:

| 指标名 | 说明 | 分桶 (ms) |
|--------|------|-----------|
| `llm.latency` | LLM 调用延迟 | 100, 500, 1000, 3000, 5000, 10000, 30000 |
| `tool.latency` | 工具执行延迟 | 10, 50, 100, 500, 1000, 5000, 30000 |
| `token.usage` | 每次 LLM 调用消耗的 token 数 | 100, 500, 1000, 5000, 10000, 50000 |

**仪表 (Gauge)**:

| 指标名 | 说明 |
|--------|------|
| `session.token.total` | 当前会话总 token 消耗 |
| `session.tool.rounds` | 当前会话工具调用轮次 |
| `session.messages.count` | 当前会话消息数 |

### 12.5 Tracing（链路追踪）

**Span 模型**:

```
Trace: {traceId} (一次 AgentLoop.run() 调用)
  │
  ├── Span: "agent.turn" (整轮对话)
  │   ├── Span: "llm.call" (第 1 次 LLM 调用)
  │   ├── Span: "tool.execute:read" (读取文件)
  │   ├── Span: "tool.execute:edit" (编辑文件)
  │   ├── Span: "tool.execute:bash" (执行命令)
  │   └── Span: "llm.call" (第 2 次 LLM 调用)
  │
  └── Span: "agent.turn" (下一轮)
```

**Span 属性**:

| 属性 | 说明 |
|------|------|
| `span_id` | 唯一 Span ID (uuid v4 短格式) |
| `trace_id` | 所属 Trace ID |
| `parent_span_id` | 父 Span ID |
| `name` | 操作名称 |
| `start_time` | 开始时间戳 (ISO 8601) |
| `end_time` | 结束时间戳 |
| `duration_ms` | 耗时毫秒 |
| `status` | ok / error |
| `metadata` | 附加信息 (模型名、工具名、token 数等) |

### 12.6 集成方式

通过 `AgentLoopContext` 注入，与 ErrorRecovery 和 Summarizer 模式一致：

```typescript
// AgentLoopContext 新增字段
interface AgentLoopContext {
    // ...现有字段
    observability?: ObservabilityManager;
}
```

Agent Loop 在关键节点调用 `observability` 的对应方法，ObservabilityManager 内部协调 Transcript / Metrics / Tracer 三个子系统。

### 12.7 配置示例

```jsonc
// .claude/settings.json
{
  "observability": {
    "transcript": {
      "enabled": true,
      "dir": "~/.y-claude-code/transcripts"
    },
    "metrics": {
      "enabled": true
    },
    "tracing": {
      "enabled": true
    }
  }
}
```
