## 目标

审查代码变更，从以下维度分析并给出具体改进建议。

## 审查维度

### 1. 正确性
- 逻辑是否正确，边界条件是否覆盖
- 是否存在空指针/undefined 访问风险
- 异步操作是否正确处理（Promise、async/await）
- 类型使用是否安全（TypeScript）

### 2. 安全性
- 是否存在注入风险（SQL、命令、XSS）
- 敏感信息是否暴露（API Key、密码硬编码）
- 路径遍历、权限校验是否到位
- 依赖是否有已知漏洞

### 3. 性能
- 是否存在不必要的重复计算/请求
- 大列表是否缺少分页/虚拟化
- 内存泄漏风险（事件监听未清理、闭包引用）
- 是否有 N+1 查询问题

### 4. 可维护性
- 函数是否职责单一、长度合理
- 命名是否清晰、符合项目约定
- 是否存在重复代码
- 错误处理是否完善

### 5. 测试
- 关键逻辑是否有测试覆盖
- 边界条件是否有测试用例
- Mock 是否合理反映真实场景

## 输出格式

```
## 审查结果

### 严重 (必须修复)
- [文件名:行号] 问题描述 → 修复建议

### 建议 (应该改进)
- [文件名:行号] 问题描述 → 改进建议

### 优化 (可选)
- [文件名:行号] 优化方向

## 总结
- 变更规模：X 文件，+Y/-Z 行
- 整体评价：...
```
